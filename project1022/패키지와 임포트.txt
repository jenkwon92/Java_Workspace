     계속 누적하다 보면 클래스 명의 중복을 피하기란 쉽지 않다. o 
(2) 클래스 명의 중복을 원천적으로 방지하기 위해서는 서로 다른 디렉토리로 구분하여 저장하면 된다. o
(3) 관련성 있는 클래스들을 디렉토리 단위로 구분하여 모아놓는 관리방법을 패키지라고 한다. o
(4) 클래스 파일명이 동일하더라도 패키지가 다를 경우엔 중복의 문제는 발생하지 않는다. o
(5) 서로 다른 디렉토리에 존재하는 클래스간의 접근은 불가능하다. x

	단, 해당 패키지에서 public으로 공개했을때만 가능 - 접근제한자
(6) 패키지화하여 관리하는 목적은 중복도 피하고, 관련성 있는 클래스들 간의 관리 목적 때문이다. o

(1) import D:/ minjava x
	D:/   ← 이 표현 자체가 윈도우 용이므로 자바의 슬로건에 위배됩니다. 자바는 특정 OS 에 의존하지 하면 안됩니다.          

(2) import minjava /pack1 x
	이유는? 자바의 패키지는 일반 디렉토리와는 달리 디렉토리 구분은 점(.)으로 합니다
               또한 클래스명이 빠져있습니다
               따라서  import minjava.pack1.클래스명;

(3) import pack1; x
	
(4) import D:/minjava/pack/Car x
(5) import pack1.Car   o
(6) import pack1.*   o

다음 설명 중 올바르지 않은 것은?
(1) 클래스를 하나의 디렉토리에 몰아넣으면 클래스 명이 중복될 경우 중복 파일 문제가 발생할 수 있다. o

(2) 관련 있는 디렉토리로 비슷한 업무의 클래스들을 묶어 놓는 것은 관리상 효율성이 있다. o

(3) 패키지 명은 소문자로 하되, 주로 개발 밴더사의 도메인을 거꾸로 기재하여 배포한다. o
	실무에서는 패키지명을 공부한 것처럼 단순하게 하지않고, 다른 회사와 협력할 떄, 서로 패키지와 클래스를 주고받는경우가 있는데
	내가 만든 클래스와 패키지명이 다른 회사 직원것과 완전 똑같을 수도 있음. 덮어써버리는것이 문제 
	이런 문제를 해결하기 위해 개발자들 사이에서는 자기회사 도메인을 거꾸로 뒤집은 패키지를 일단 상위 패키지로둠
	test.com 이라는 도메인을 보유한 회사라면  com.test.study 이런식으로 뒤집음

(4) 같은 패키지 내에 있는 클래스들 끼리는 언제든지 서로의 데이터를 접근할 수 있다. x
	보안이 뚫리는 경우가 있으므로 , 접근제한자로 제한을 둘 수 있음.
	언제든지라는 표현이 잘못됨
